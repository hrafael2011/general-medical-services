# Spec 09 - Asistente Operacional y Tools

## Goal

Define how the Telegram assistant interprets natural language requests and answers operational questions using internal tools, structured data, and controlled explanations.

The assistant must be flexible in language but strict in truth source. It should understand many ways of asking the same thing without requiring a database of pre-written questions.

Developer-facing prompts, tool names, endpoint names, and logs may be written in English for maintainability and portfolio clarity. User-facing assistant responses must be written in Spanish.

## Core Principle

The LLM interprets. The backend decides.

The assistant must:

- classify the user's intent,
- extract entities such as dates, doctors, areas, departments, and criteria,
- call an approved internal tool or endpoint,
- respond only from returned tool data,
- ask for clarification when required entities are missing,
- reject out-of-domain requests,
- and log every operational interaction.

The assistant must not:

- answer from model memory,
- invent doctors, assignments, history, availability, rules, or scores,
- query raw database tables directly,
- make autonomous operational decisions without backend validation,
- or treat RAG/document retrieval as the primary source for structured operational truth.

## Intent Model

The system uses a catalog of supported capabilities, not a list of every possible user question.

Examples of general capabilities:

- doctor lookup,
- availability lookup,
- doctor history,
- workload ranking,
- monthly service target and maximum lookup,
- calendar status,
- fairness conflict lookup,
- candidate recommendation,
- decision explanation,
- mission assignment with confirmation,
- notification summary,
- report/export generation,
- audit lookup.

The user may phrase requests naturally, with typos, incomplete wording, or informal language. The LLM maps the request to the closest supported intent when confidence is sufficient.

If the request is ambiguous, the assistant asks one focused clarification question.

If no supported intent exists, the assistant must reject or fallback without hallucinating.

## Tool-Calling Flow

Runtime flow:

1. Telegram receives a message from a linked and active user.
2. The backend verifies the user's role and account status.
3. The LLM classifies intent and extracts entities.
4. The orchestrator validates that the intent is allowed for the user's role.
5. If required entities are missing, the assistant asks for clarification.
6. If valid, the orchestrator calls the mapped internal tool or API endpoint.
7. The backend tool queries and calculates from canonical data.
8. The LLM formats the response using only the tool result.
9. The interaction is persisted for audit and improvement.

For MVP, Telegram assistant actions are read-only from an operational data perspective except for confirmed mission assignment.

Telegram may query, recommend, explain, and generate/send reports or exports.

Telegram must not create, modify, approve, or deactivate regular service calendar records, doctors, users, availability, restrictions, or rules.

The only MVP write action allowed through Telegram is confirming a mission assignment after the assistant proposes candidates and the encargado explicitly confirms.

## Tool Contract

Every assistant tool must define:

- `tool_name`
- `intent_ids`
- `allowed_roles`
- `required_entities`
- `optional_entities`
- `input_schema`
- `output_schema`
- `freshness_policy`
- `cache_policy`
- `audit_level`
- `fallback_behavior`

Tool output should be structured enough for both machine validation and natural language explanation.

Tool schemas should use English field names. Human-readable response text generated from tool results must be Spanish.

## Standard Response Outcomes

Supported request with data:

- answer using the tool result,
- include concise rationale when the answer depends on ranking, recommendation, or fairness,
- avoid exposing internal implementation details unless requested by an authorized role.

Report/export request with data:

- generate or retrieve the requested report through an approved backend tool,
- send the generated file through Telegram when supported,
- record file generation and delivery metadata for audit.

Supported request without data:

- "No tengo esa informacion en el sistema."

Out-of-domain request:

- "No puedo responder eso porque esta fuera del alcance del sistema."

Ambiguous request:

- ask a focused clarification question, such as date, range, doctor, area, or department.

Unauthorized request:

- "No tienes permiso para consultar esa informacion."

## Recommendation Responses

Recommendations must be generated by deterministic backend logic using canonical data. The LLM may explain the result but must not calculate or alter ranking by itself.

Recommendation tool output must include:

- candidate doctor,
- eligibility status,
- score,
- reasons in favor,
- warnings,
- hard blockers,
- data used,
- rules applied,
- `data_as_of`.

Examples of data used:

- recent services,
- monthly workload,
- configured monthly service target and maximum,
- service area history,
- availability,
- licenses,
- restrictions,
- mission eligibility when relevant,
- fairness evaluations.

If no eligible candidates exist, the response must explain the blockers or unresolved gaps returned by the backend.

## Mission Requests Through Telegram

Telegram may support mission assignment as a controlled write action in MVP.

Required mission request entities:

- mission date,
- participant count.

Optional mission request entities:

- location,
- description or motive.

Flow:

1. Encargado asks for mission candidates with date and count.
2. If date or count is missing, assistant asks a focused clarification question.
3. Backend uses the stored monthly mission ranking and filters candidates by date, mission eligibility, active blocks, service conflicts, availability, and spacing.
4. Doctors already assigned to a mission in the same month are lowered in priority, not automatically excluded.
5. Doctors occupied with service on the requested date are skipped for the primary proposal.
6. Assistant returns the proposed candidates with concise reasons and warnings.
7. Encargado may ask follow-up questions or request alternate candidates.
8. Alternate candidates may include warning candidates, including occupied or lower-priority doctors, but hard-blocked doctors remain excluded.
9. Assistant asks for explicit confirmation before writing the mission assignment.
10. After confirmation, backend records the mission, participants, rationale, audit event, and notification jobs.

Mission confirmation must be explicit. The assistant must not infer confirmation from an ambiguous response.

## Decision Explanations

When the user asks why a doctor was selected, skipped, or ranked, the assistant must call an explanation tool.

The explanation must be based on stored assignments, fairness evaluations, rule results, overrides, and audit records.

The assistant must distinguish between:

- a generated decision,
- a manual override,
- an imported historical assignment,
- and unavailable explanation data.

If no explanation data exists, respond:

- "No tengo esa informacion en el sistema."

## Cache Policy

The assistant and backend may use cache to reduce unnecessary database calls, but freshness must be explicit for operational answers.

Allowed cache categories:

- request-scope cache for repeated reads during one interaction,
- short TTL cache for stable catalogs and frequent summaries,
- invalidation-based cache for records changed by operational actions.

Good candidates for short TTL cache:

- active doctors list,
- departments,
- ranks,
- service areas,
- active fairness rules,
- official calendar summary for a month.

Avoid aggressive caching for:

- recommendations,
- availability decisions,
- audit queries,
- notification delivery status,
- manual overrides,
- security or permission checks.

Invalidation triggers:

- doctor created or updated,
- availability changed,
- license created or updated,
- restriction changed,
- calendar approved or modified,
- assignment changed,
- fairness rule changed,
- user role or account status changed.

Operational tool responses should include internal freshness metadata:

- `data_as_of`
- `cache_status`
- `ttl_seconds` when applicable

The assistant does not need to show cache metadata to the end user unless useful or requested by an authorized role.

## RAG Policy

RAG may be added later for documents, institutional policies, manuals, or historical notes that are not yet structured.

RAG must not be the primary authority for:

- doctor identity,
- current availability,
- assignments,
- licenses,
- restrictions,
- fairness scores,
- recommendations,
- audit records,
- permissions.

If a retrieved document conflicts with canonical database data, canonical database data wins for operational answers.

## Security Requirements

Every assistant request must validate:

- Telegram user is linked,
- matched system user is active,
- user has completed required password change,
- user role is authorized for the intent,
- requested data is within the role's permission scope.

Sensitive or state-changing actions require two-step confirmation and audit.

In MVP, the only state-changing Telegram action is mission assignment confirmation. Other state-changing Telegram actions are reserved for future phases.

Prompt injection attempts must not override system rules, tool policies, permissions, or fallback behavior.

## Audit Requirements

Every Telegram interaction must persist:

- Telegram user id,
- matched user id,
- role,
- input text,
- detected intent,
- extracted entities,
- confidence where available,
- tool called,
- tool request,
- tool response summary or reference,
- response text,
- cache status where relevant,
- fallback reason,
- status,
- created timestamp.

Critical assistant actions must also create an `AuditEvent`.

## Acceptance Criteria

1. Given different phrasings of the same supported operational request, when the user asks through Telegram, then the assistant maps them to the same supported intent without requiring pre-written question entries.
2. Given a supported request with missing required entities, when the assistant cannot safely call the tool, then it asks one focused clarification question.
3. Given a supported request with no available data, when the tool returns an empty result, then the assistant responds "No tengo esa informacion en el sistema."
4. Given an out-of-domain request, when no supported intent applies, then the assistant rejects it without hallucinating.
5. Given a recommendation request, when candidates are returned, then the response explains the ranking using only backend-provided reasons, scores, rules, and data freshness.
6. Given a user without permission for an intent, when they ask for restricted information, then the assistant denies the request and logs the event.
7. Given repeated reads in one assistant interaction, when the same supporting data is needed more than once, then request-scope cache avoids duplicate database calls without changing the answer.
8. Given a doctor, availability, license, restriction, calendar, assignment, rule, or user permission change, when cached data exists, then affected cache entries are invalidated or expire according to policy.
9. Given an authorized report request through Telegram, when the requested report is within system scope, then the assistant generates or retrieves the report and sends it through Telegram.
10. Given a Telegram request to modify or approve regular service, doctor, user, availability, restriction, or rule data in MVP, then the assistant rejects the action and directs the user to the panel.
11. Given an encargado requests mission candidates with date and count, then Telegram proposes candidates from the stored mission ranking after applying date-specific filters.
12. Given the encargado explicitly confirms proposed mission candidates, then Telegram may create the mission assignment, audit the action, and trigger notifications.
